# Full Project Audit — 2025-05-26

Comprehensive audit of TradeJournal v3 covering backend routers, services, frontend hooks/stores/components, schemas, and cross-cutting concerns.

---

## CRITICAL — Data Integrity & Security

### R1. No Authentication on Any Endpoint (Except `/auth/`)

All routers except `auth.py` have zero auth checks. Unauthenticated users can CRUD trades, capital, journal entries, etc.

**Files:** Every router in `backend/app/routers/` except `auth.py`

**Severity:** CRITICAL — security vulnerability

**Fix:** Add `Depends(get_current_user)` to all mutation endpoints, and read endpoints that expose user data. Group into public vs. authenticated routes.

---

### R2. `_auto_set_status` Not Called After Several Mutation Paths

Imported/closed trades retain `status="open"` because `_auto_set_status()` is only called in `trades.py` router, not in service layer.

**Missing calls:**

| Path | File:Line | 
|------|-----------|
| `TradeService.merge_or_create` | `trade_service.py:70-76` |
| `TradeService._merge_trade` | `trade_service.py:98-148` |
| `TradeService.merge_duplicates` | `trade_service.py:183-224` |
| `broker_import.py` import | `broker_import.py:248` |
| `dhan.py` sync | `dhan.py:48` |
| `trade_ideas.py` convert_to_trade | `trade_ideas.py:117-140` |

**Severity:** CRITICAL — wrong trade status breaks all status-based queries

**Fix:** Add `_auto_set_status()` call in `TradeService.merge_or_create` after trade creation/merge, and in broker import / idea conversion paths.

---

### R3. `_update_setup_stats` Not Called After Several Mutations

Playbook stats (trade_count, win_rate, avg_r) go stale after mutations that skip stats recalculation.

**Missing calls:**

| Path | File |
|------|------|
| `broker_import.py` import | After creating trades |
| `dhan.py` sync | After creating trades |
| `trade_ideas.py` convert_to_trade | After creating trade |
| `partial_exit.py` create | After partial exit (changes PnL → changes win rate) |
| `partial_exit.py` delete | After deleting partial exit |

**Severity:** HIGH — stale analytics

**Fix:** Call `_update_setup_stats(db, setup_name)` after each mutation that changes PnL or creates/deletes trades.

---

### R4. `_auto_reconcile` Not Called After Several Mutations

Capital balance goes stale when mutations skip reconciliation.

**Missing calls:**

| Path | File |
|------|------|
| `dhan.py` sync | After creating/updating trades |
| `trade_ideas.py` convert_to_trade | After creating trade |
| `trades.py` create_stop_history | After changing stop_price (affects deployed capital / heat) |

**Severity:** HIGH — incorrect account balance

**Fix:** Call `_auto_reconcile(db)` after each capital-impacting mutation.

### R5. `_reconcile_account` Commits Mid-Transaction

`_reconcile_account()` in `capital_events.py:78` calls `db.commit()` internally. When called from `trades.py`, `partial_exit.py`, or `broker_import.py`, this commits reconciliation before the outer transaction completes. If the outer `db.commit()` fails, the reconciliation is already persisted — partial state.

**Files:**

- `capital_events.py:78` — `db.commit()` inside `_reconcile_account()`
- `trades.py:155-157` — outer commit after reconcile
- `partial_exit.py:97-99` — outer commit after reconcile

**Severity:** HIGH — transactional inconsistency

**Fix:** Remove `db.commit()` from `_reconcile_account()`. Let the caller commit. All callers must call `db.commit()` once after all mutations + reconciliation.

---

### R6. IST Timestamps Stored in DB Instead of UTC

At audit time, active docs described trade datetimes as UTC, but 6+ places stored IST-local times without timezone info:

| File | Line | Pattern |
|------|------|---------|
| `emotion_log.py` | 42 | `datetime.now(EL_IST).replace(tzinfo=None)` |
| `capital_events.py` | 72, 77, 115, 260, 284 | `datetime.now(CE_IST).replace(tzinfo=None)` |
| `trades.py` | 457 | `datetime.now(TRADE_IST).replace(tzinfo=None)` |
| `trade_timeline.py` | 35 | `datetime.now(TL_IST).replace(tzinfo=None)` |

**Severity:** HIGH — double-conversion on display

**Fix:** Replace all `datetime.now(IST).replace(tzinfo=None)` with `datetime.now(timezone.utc).replace(tzinfo=None)`.

---

### R7. `capital_events.py` Update/Delete Use Manual Delta Instead of Full Reconcile

`update_capital_event` (line 235-266) and `delete_capital_event` (line 269-290) manually adjust `current_balance` by the delta instead of calling `_reconcile_account()`. If the stored delta was stale or event type changed, the manual adjustment produces wrong balances.

**Files:**

- `capital_events.py:235-266` — update uses manual delta
- `capital_events.py:269-290` — delete uses manual reverse

**Severity:** HIGH — balance drift

**Fix:** Replace manual delta logic with `_reconcile_account(account_id, db)` call.

---

### R8. `TradeService.merge_or_create` Commits Internally

`merge_or_create` in `trade_service.py:73,89` calls `self.db.commit()` internally. The calling router in `trades.py:create_trade` then adds timeline entries and commits again. If timeline creation fails after `merge_or_create` succeeds, the trade is committed but timeline is lost.

**Files:**

- `trade_service.py:73,89` — internal `db.commit()`

**Severity:** HIGH — orphaned trade records

**Fix:** Remove `db.commit()` from `merge_or_create`. Let the caller manage the transaction. Add `db.flush()` instead to get the trade ID without committing.
